## Parameter Insights

- **Plain ORB breakout broad survivors did not clear strict exact cost/slippage stress before holdout**: The ORB Futures Surface v1 workflow found four one-sided exact-replay pass rows on pre-holdout/discovery (`NQ Asia 5m long`, `GC Asia 5m long`, `RTY LDN 15m long`, `CL LDN 30m short`) after the broad grid and live-style exact replay. The next frozen exact stress applied `2x` baseline commission plus `2` adverse ticks per side to the exact replay trade ledger over `2021-01-01` to `2024-12-31`, with holdout still closed. No row fully passed. NQ Asia stayed positive but dropped to `+33.76R`, PF `1.15`, with 2023 negative (`-3.89R`), so it is WATCH only. GC Asia stayed positive but dropped to `+15.45R`, PF `1.09`, with 2021/2022 negative (`-4.81R` / `-2.88R`), also WATCH only. CL LDN failed (`-13.39R`, PF `0.88`) and RTY LDN failed (`-0.14R`, PF `1.00`). Practical rule: do not open holdout for plain ORB Futures Surface v1 as-is. If revived, start from NQ Asia or GC Asia and improve friction robustness before any final-test use. References: `backtesting/learnings/reports/ORB_FUTURES_SURFACE_V1_EXACT_REPLAY_20260618.md`, `backtesting/learnings/reports/ORB_FUTURES_SURFACE_V1_EXACT_STRESS_20260618.md`.
- **Chasing newer/higher continuation gaps is a hard NO-GO for the current `ALPHA_V1` ORB book**: A focused trailing-two-year test on `2024-04-17` to `2026-03-24` changed same-day continuation/reversal FVG selection from the live/default `first` rule to an `extreme` chase rule (highest bullish gap / lowest bearish gap of the session-day). The chase rule did improve fill rate, but it destroyed expectancy across all three active ORB legs: `NQ Asia` went from `135` filled trades / `+56.2R` to `136` / `-4.5R`; `ES Asia` from `275` / `+41.5R` to `287` / `-164.8R`; `ES NY` from `177` / `+22.9R` to `183` / `-39.0R`. Combined ORB total moved from `+120.7R` to `-208.3R` for only `+19` extra fills. The regime read did not rescue the thesis: `NQ Asia` damage narrowed inside bull regimes but still stayed worse than baseline even in `bull_high_vol`, while both ES legs remained decisively negative inside the exact bull/high-vol buckets that were supposed to be the "expansive" sweet spot. Practical rule: keep `continuation_fvg_selection="first"` for the current ORB families; if a chase-style entry idea is revisited, it needs a different setup definition rather than simply ratcheting the same-day valid gap higher. Reference: `backtesting/learnings/reports/ALPHA_V1_ORB_GAP_CHASE_20240417_20260324.md`.
- **max_gap_points=100 is too loose for ES**: NQ-optimized config applied to ES (Run #16) produced Sharpe 0.92 vs NQ's 1.39 over comparable 10yr period. ES has smaller absolute point moves than NQ, so 100-point max gap filter lets in low-quality setups. Needs its own sweep.
- **Best NQ config (Runs 9-14)**: rr=2.5, tp1_ratio=0.5, ny_stop_atr_pct=7.5, ny_min_gap_atr_pct=2.25, ny_max_gap_points=100, asia_stop_atr_pct=5.25, asia_min_gap_atr_pct=0.9, asia_max_gap_points=50. Sharpe 2.86 on 2024-2025, degrades to 1.39 on 2016-2026.
- **NQ_NY_LSI medium-vol gate does not transfer cleanly to FAST_V1.1 params**: Reusing the RR2/TP0.5 regime-avoidance rule (`skip bull_medium_vol + sideways_medium_vol`) on FAST_V1.1 NQ_NY_LSI (`rr=3.0`, `tp1_ratio=0.34`, `atr_length=10`, Wed+Thu excluded) improved trade quality but hurt risk-adjusted return on a Calmar basis. On 2016-01-01 to 2026-03-31, trades fell 611→446, WR rose 59.3%→62.1%, PF rose 1.61→1.84, Sharpe rose 3.22→4.14, but Net R fell +120.1R→+113.9R and Calmar fell 18.10→17.18 with unchanged DD (-6.63R). On 2021-01-01 to 2026-03-31, the same pattern held: 312→211 trades, PF 1.70→1.96, Sharpe 3.44→4.28, but Net R +65.9R→+55.9R and Calmar 9.94→8.43. Conclusion: this gate is parameter-dependent; it helps the RR2/TP0.5 variant more than the FAST_V1.1 RR3/TP0.34 variant.
- **NQ NY ALPHA_V1 LSI is stronger as an explicit entry-gated non-consumptive variant than as the corrected "fresh sweep" version**: An April 8, 2026 retest compared the corrected branch (`08:30-14:30` sweep window, stale pivots consumed on any breach) against a live-matched legacy branch where swings are tracked continuously but sweeps only count once the entry window opens and pre-entry breaches do not retire the pivot. On `2016-01-01` to `2025-12-31`, corrected all-days did `319` trades, `+35.5R`, Calmar `5.01`; the live-matched legacy branch did `544` trades, `+62.8R`, Calmar `8.58`. With `Wed+Thu` excluded, corrected did `180` trades, `+31.9R`, Calmar `7.21`, while legacy live did `310` trades, `+59.8R`, Calmar `10.67` with `0` negative years. `n_left=8` remained best in both branches. Treat this as a separate strategy thesis, not a bug to leave implicit, because live restart recovery only warms same-day bars while continuous runtime preserves older swing levels.
- **Exact execution replay disagrees with the research surrogate on weekday cuts for `legacy-LSI`**: Once `legacy-LSI` was made explicit in the live engine and replayed directly as a single-leg `NQ_NY_LSI` profile (`rr=3.0`, `tp1=0.34`, `gap=5.0`, `n_left=8`, `n_right=60`, all days), the exact replay on `2016-01-01` to `2025-12-31` printed `599` trades, PF `1.47`, Sharpe `2.54`, Net R `+83.0`, DD `-5.51R`, Calmar `15.06`, with every full year positive. On that exact replay trade set, excluding `Thu` reduced Calmar to `13.05`, excluding `Wed+Thu` to `10.62`, and excluding `Thu+Fri` to `6.48`. Conclusion: for this branch, use exact execution replay as the tie-breaker on weekday policy and start from all days enabled.
- **A direct exact single-leg head-to-head still favors legacy `ALPHA_V1` LSI over `HTF_LSI_5M_LAG24` as the all-weather account-farming branch**: Using the same exact replay path and the same payout models on the latest common end `2026-03-24`, the true `ALPHA_V1` single-leg config (`legacy-LSI`, `09:35-15:30`, `rr=3.0`, `tp1=0.34`, `gap=5.0`, `n_left=8`, `n_right=60`, `fvg=20/5`, `Wed+Thu` excluded) beat the HTF branch used in that exact comparison on long-run robustness even though HTF traded more often. On `2016-01-01` to `2025-12-31`, legacy printed `427` trades, PF `1.595`, avg R `0.189`, total R `+80.67`, DD `-6.36R`, Calmar `12.68`, funded payout `81.6%`, prop payout `93.9%`; HTF lag24 printed `530` trades, PF `1.357`, avg R `0.165`, total R `+87.20`, DD `-12.97R`, Calmar `6.72`, funded payout `52.0%`, prop payout `68.3%`. The important nuance is recency: `2024` favored HTF decisively while legacy was roughly flat, and `2025` split with legacy better on raw PF/DD but HTF better on trade count, total R, and funded payout conversion. Conclusion: legacy remains the better core live leg for long-run payout farming, while HTF lag24 is the newer alternate that handled the post-2023 regime mix better than legacy’s weak `2024`. Reference: `backtesting/learnings/reports/NQ_NY_LSI_LEGACY_VS_HTF_EXACT_COMPARISON.md`.
- **`HTF_LSI_5M_LAG24` is the best true / canonical LSI candidate even though `legacy-LSI` still wins the broader head-to-head**: The only branch that currently beats HTF lag24 on long-run exact replay is `legacy-LSI`, but that branch depends on the old broken entry-gated, non-consumptive sweep semantics and should be treated as a separate non-canonical strategy bucket rather than the benchmark for “real” LSI. Practical rule: if the question is “what is the best actual LSI branch to build from,” the answer is `HTF_LSI_5M_LAG24`; if the question is “what old live-like branch still farms payouts best,” the answer can still be legacy.
- **Removing the research-side HTF session trade cap did not improve on `HTF_LSI_5M_LAG24`; it mostly made the lower timeframes noisier**: A fresh uncapped rerun set `htf_trade_max_per_session=0` and re-ran the `1m/2m/3m/5m` HTF-LSI transfer + lag packet. `5m` was the key result: uncapping changed nothing. The best uncapped lag row was still `long / fvg_limit / lag24`, with the exact same validation PF / avg R / Calmar as the capped study (`1.597 / 0.268 / 6.38`, `127` validation trades). The lower timeframes did not become cleaner. `3m` shifted to an uncapped `lag19` winner with better validation PF but lower validation Calmar than the capped `lag0` row and still negative discovery avg R, so robustness worsened. `2m` uncapped produced a flashy both-direction `lag1` row, but it was only `12` validation trades. `1m` uncapped broadened to `both / fvg_limit / lag0`, but the capped `long / close / lag10` challenger still had materially better validation PF / avg R / Calmar (`1.888 / 0.420 / 4.91` vs `1.279 / 0.132 / 3.76`). Takeaway: uncapping does not weaken the `5m lag24` conclusion; it strengthens it by showing the best `5m` branch is stable even when the lower-timeframe variants get more sample-fragile. Reference: `backtesting/learnings/reports/NQ_NY_HTF_LSI_UNCAPPED_RERUN.md`.
- **On the frozen NQ NY `2m` HTF-LSI anchor, broadening sweep sources from unswept `1h` pivots to completed session/day/week reference levels adds trades but does not improve edge quality**: A focused pre-holdout compare on the frozen `2m` anchor (`long`, `fvg_limit`, `08:30-15:00`, `rr=3.0`, `tp1=0.6`, `gap=3.0`, `htf60 n3`, `cap1`, `left50`, `right5`, `lag0`) tested three variants: `htf_only`, `reference_only`, and `htf_plus_reference`, using the basket `new_york_high/low`, `asia_high/low`, `london_high/low`, `previous_day_high/low`, `previous_week_high/low`. Important nuance: on this frozen `long` branch, only the **low-side** published levels actually contributed fills, led by `asia_low`, `london_low`, and `new_york_low`; the added highs did nothing. Practical rule: if the goal is the best base config for future confluence testing, keep the honest `2m` `htf_only` anchor. If the goal later becomes count expansion, `htf_plus_reference` is the only side branch worth keeping alive, but treat it as a weaker higher-count variant rather than an upgrade. Reference: `backtesting/learnings/reports/NQ_NY_HTF_LSI_2M_SWEEP_SOURCE_COMPARE.md`.
- **On that same frozen NQ NY `2m` HTF-LSI anchor, adding same-day `data_high/data_low` spike-candle levels is a clear downgrade, not a useful count expansion**: A second pre-holdout compare defined `data_high/data_low` as a completed `1m` candle with range >= `15%` of previous-day ATR, published on the first eligible base bar after the `1m` close and kept alive for the rest of that day. Three variants were tested: `htf_only`, `data_only`, and `htf_plus_data`. The original `htf_only` anchor again remained best on stitched OOS at `486` trades, PF `1.212`, avg R `0.104`, Calmar `3.76`, DD `-13.41R`. `data_only` failed outright (`585` trades, PF `0.984`, avg R `-0.010`, Calmar `-0.21`, DD `-25.98R`). `htf_plus_data` added a lot of traffic (`797` stitched-OOS trades, `1222` pre-holdout filled trades) but materially weakened the branch to PF `1.087`, avg R `0.043`, Calmar `1.52`, DD `-22.74R`, with validation already effectively dead at PF `0.994`, avg R `-0.003`, Calmar `-0.06`. - **For exact/live HTF-LSI sizing, `max_open_contracts=20` is effectively uncapped in the recent sample while `1` is a real distortion**: A direct exact replay comparison on `HTF_LSI_5M_LAG24` held the trade stream fixed and only changed the execution profile’s contract ceiling: `cap=1`, `cap=20`, and true uncapped `cap=0`. `cap=1` forced every trade to `1` MNQ and produced the most distorted sizing profile. `cap=20` lifted the full-window average size to `6.76` MNQ (median `4`, max `20`) and changed the exact `10Y` profile to PF `1.365`, total R `+85.33`, DD `-10.75R`, versus `1.357`, `+87.20`, `-12.97R` at `cap=1`. More importantly, `cap=20` and true uncapped `cap=0` were effectively the same on `2024`, `2025`, and `2026 YTD`: same trade counts, same avg R / total R / max DD, because recent desired size never exceeded `10` MNQ. Takeaway: for current HTF-LSI exact/live work, treat `max_open_contracts=20` as the practical uncapped default; the old `1`-contract profile was an artificial choke collar rather than a meaningful risk setting. - **If `HTF_LSI_5M_LAG24` is used as the `ALPHA_V1` NQ NY replacement, the best recent-year sprint size is `$300`, not `$400`**: An exact replay risk sweep compared `$250/$300/$350/$400/$450/$500` on `2024-01-01` to `2026-03-24` using the same `14`-day stagger, `+$2500 / -$2000` prop-account math used in `ALPHA_V1`. The clean compromise was `$300`: recent prop payout `98.0%`, breach `2.0%`, average payout `176d`, EV/start `+$2410`. `$250` was the safest row (`100%` payout, `0%` breach, `193d`) but too slow to be the practical swap-in, while `$350+` degraded too fast (`$350`: `94.1%` payout, `5.9%` breach; `$400`: `92.2%`, `7.8%`; `$500`: `78.4%`, `21.6%`). A `10Y` exact sanity check kept the same shape: `$250` best on pure payout probability but very slow (`94.5%`, `392d`), `$300` the better balance (`90.6%`, `304d`), `$400` materially weaker on account outcomes (`76.8%`, `158d`). Takeaway: for an `ALPHA_V1`-style sprint account, use `risk_usd=300` as the recommended HTF-LSI replacement size; keep `$250` as the extra-conservative fallback and avoid `$350+` unless faster payout is worth the meaningfully worse breach profile. References: `backtesting/data/results/nq_ny_htf_lsi_replacement_risk_recent.json`, `backtesting/data/results/nq_ny_htf_lsi_replacement_risk_10y.json`.

- **Inside the swapped `ALPHA_V1` portfolio, the `2024-2025` payout-speed frontier favors `$350` over the initial `$300` replacement size**: A direct exact sweep varied only the HTF-LSI replacement leg’s risk inside the combined portfolio on `2024-01-01` to `2025-12-31`. The curve was not monotonic. Starting from `$300`, the combined portfolio printed `82.7%` payout with `100.3d` average payout. `$325` preserved the same payout rate and cut the average to `93.2d`, but the best balanced step was `$350`, which improved both dimensions simultaneously to `84.6%` payout and `86.9d` average payout. `$400` cut the average further to `72.9d` while still holding the same `82.7%` payout rate as `$300`. - **A full exact four-leg frontier says the swapped `ALPHA_V1` portfolio still forces a real speed-vs-success choice, and the offline blended frontier was too optimistic**: After varying all four legs together on `2024-01-01` to `2025-12-31` (`HTF_LSI`, `NQ_Asia`, `ES_Asia`, `ES_NY`), the exact combined-profile verify materially downgraded the fast blended candidates. The best verified payout-quality row was still the baseline-style mix `HTF_LSI=300 / NQ_Asia=300 / ES_Asia=200 / ES_NY=300`, which printed `84.3%` payout and `109.5d` average payout. The fastest verified mix that still held roughly an `80%` payout rate was `HTF_LSI=400 / NQ_Asia=250 / ES_Asia=250 / ES_NY=350`, which improved to `80.8%` payout and `66.0d` average payout. Reference: `backtesting/data/results/alpha_v1_htf_portfolio_frontier_2024_2025.json`.
- **On the swapped `ALPHA_V1` portfolio, R-triggered account starts beat fixed-time staggering for the exact first-payout tradeoff**: A dedicated stagger-policy frontier compared `7d / 10d / 14d / 21d` calendar staggers against a master-stream R-triggered model that starts new accounts whenever the combined portfolio moves by `2R / 3R / 4R / 5R`, with `1R = $500` so the trigger logic matches the same `+5R / -4R` payout model. The fixed-time winners all landed near the same aggressive mix and capped out around `80-81%` payout with `60-64d` average payout: `time_14d` with `400 / 250 / 250 / 400` printed `80.8%` and `60.0d`, `time_7d` `80.8%` and `61.1d`, and `time_10d` `80.8%` and `62.2d` with `ES_NY=350`. The R-triggered model was better almost across the board. The same core aggressive mix `400 / 250 / 250 / 400` printed `86.5%` payout and `55.8d` at `2R`, `86.4%` and `56.0d` at `3R`, `86.7%` and `58.8d` at `4R`, and `84.6%` and `50.6d` at `5R`. Takeaway: if the user wants the fastest payout engine that still keeps a strong payout percentage, prefer **R-triggered staggering** over the old fixed `14d` clock; `r_trigger_2R` is the best balanced row, while `r_trigger_5R` is the faster aggressive version. Reference: `backtesting/data/results/alpha_v1_htf_stagger_policy_frontier_2024_2025.json`.
- **Adding a minimum spacing floor on top of `r_trigger_2R` was only a tiny tuning aid, not a new default**: A follow-up hybrid test checked whether `2R` starts should also obey a minimum calendar gap of `3d / 5d / 7d / 10d`. The answer was mostly no. On the aggressive `400 / 250 / 250 / 400` mix, plain `r_trigger_2R` printed `83.0%` payout and `56.0d` average payout; `min 3d` trimmed that to `55.2d` at the same payout rate, and `min 5d` to `55.6d`, but the gains were tiny. `min 7d` and `min 10d` both degraded the frontier. On the safer baseline-style `300 / 300 / 200 / 300` mix, `min 7d` improved payout rate (`87.9%` vs `85.7%`) but slowed average payout (`89.3d` vs `86.7d`). Takeaway: keep plain `r_trigger_2R` as the honest operating default; only revisit a very light `min 3d` or `min 5d` floor if the real concern is start clustering rather than first-payout speed itself. Reference: `backtesting/data/results/alpha_v1_htf_hybrid_2r_min_spacing_test_2024_2025.json`.
