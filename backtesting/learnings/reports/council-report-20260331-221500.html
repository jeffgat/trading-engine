PBO/DSR Implementation — Where in the Strategy Workflow?

Should PBO (Probability of Backtest Overfitting) via CSCV and DSR (Deflated Sharpe Ratio) be implemented as a step in the trading strategy discovery workflow? Where should it fit in the 7-step process?

The Recommendation

Phase 1 — This Week

Ship PSR + DSR as annotations on every config exiting walk-forward. Cluster configs by trade overlap for effective independent trial count. ~30 lines of numpy each.

Phase 2 — Next Sprint

Modify walk-forward to retain the full config × fold OOS performance matrix. Store as numpy array per pipeline run. This is the prerequisite for CSCV.

Phase 3 — After Phase 2

Build CSCV/PBO. Run at pipeline end. PBO > 0.50 = warning. PBO > 0.70 = blocks promotion. Dynamic split count based on min 50 trades per split.

Where it fits: Step 5b — a validation gate between discovery (step 5) and phase-one robust pipeline (step 6). PSR/DSR annotate individual configs. CSCV evaluates the entire search process. Neither is a "step" that produces new candidates — they're quality gates that can block or flag.

Replace the "heuristic" caveat after Phase 1 ships. What you already compute (fold-level OOS consistency + stability score) is not heuristic — it's quantified. Add PSR/DSR values to complete the picture.

The One Thing to Do First

Modify the walk-forward optimizer to save the full config × fold OOS performance matrix instead of discarding it. Everything else — DSR, PSR, CSCV — is formula work. But without the matrix, CSCV is impossible, and you'll have to re-run every pipeline to get it. Retain the data now; do the math later.

Where the Council Agrees

  • DSR/PSR are trivially cheap (15-30 lines) and should ship immediately
  • CSCV is fundamentally different — evaluates the search process, not individual configs
  • Current system already does most anti-overfitting work through 28-fold WF and stability
  • The "heuristic" caveat is more of a labeling problem than a capability gap

Where the Council Clashes

  • Where CSCV belongs: Inline gate (Expansionist) vs end-of-pipeline (Outsider) vs parallel infrastructure (Contrarian). Peer reviews resolved: runs at end, evaluates the full search.
  • Whether to expand search: Expansionist says PBO enables wider sweeps. Three reviewers flagged this as inverted — more configs mechanically increases PBO.
  • Regime vs overfitting: First Principles wants to skip PBO for regime detection. Two reviewers called this a false dichotomy — both matter.

Blind Spots the Council Caught

  • Sample size per split: Many configs produce <30 trades per combinatorial split, making CSCV noisy
  • Config correlation: Raw trial count over-penalizes DSR. Need effective independent trial count via clustering
  • WF config matrix discarded: The real engineering bottleneck — CSCV is impossible without retaining per-config per-fold data
  • Compute cost unspecified: 4,000 configs × 28 folds × combinatorial splits is non-trivial
The Contrarian — "PSR first. PBO is parallel infrastructure, not a step."

The fatal flaw is sequencing. CSCV evaluates the entire search process, not individual candidates. If PBO comes back at 0.6 after promoting 8 finalists, what do you do? CSCV requires re-running parameter search across combinatorial splits — parallel infrastructure, not a step. DSR has a trap: 1,000-4,000 configs are NOT independent — they're correlated parametric variations. DSR will be too conservative. Walk-forward + stability does 80% of anti-overfitting work. Remaining 20% is forward paper trading. Implement PSR first.

Peer review consensus: Strongest on the nature of CSCV. Correct that correlated configs break naive DSR.

The First Principles Thinker — "Wrong failure mode. Regime change kills accounts."

PBO assumes the problem is selecting the wrong config from a large sweep. But WF with 28 folds and stability analysis already addresses that. What actually kills accounts is regime change. Don't build PBO/DSR as a workflow step. Replace the heuristic caveat with quantified versions of what you already compute. If you must implement DSR, it's 15 lines — annotation, not gate.

Peer review consensus: Identified as having the biggest blind spot — false dichotomy between regime change and overfitting. Both matter. Overfit strategies fail faster under regime change.

The Expansionist — "Gate Step 6. Enables wider search."

Massive opportunity. Fit between Step 5 and Step 6 as a gate. Rigorous PBO scores let you widen exploration space. With PBO < 0.40, sweep 10,000+ configs. DSR directly consumes trial counts. 200-400 lines of numpy.

Peer review consensus: Flagged as having inverted logic — more configs mechanically increases PBO. Cannot use PBO as a license to search more aggressively.

The Outsider — "Gates, not steps. Run at the end."

These tests aren't steps — they're gates. A step produces output; a gate kills things. If you slot PBO in the middle, people optimize around it. Run after entire pipeline, as final go/no-go. Use $50-150 account cost to set thresholds mathematically. PBO of 0.15 doesn't mean strategy works — just means you probably didn't overfit.

Peer review consensus: Strong framing on gates vs steps. Cost-math threshold approach praised.

The Executor — "Ship DSR Monday. CSCV needs WF engine changes."

DSR is 20 lines of numpy. PSR is one formula. Ship both immediately. CSCV needs the per-fold, per-config performance matrix, which WF currently throws away. Three phases: (1) DSR+PSR now, (2) Modify WF to retain matrix, (3) Build CSCV on matrix. Validation gate after WF.

Peer review consensus: Only advisor to identify the real engineering bottleneck (discarded config matrix). Phased plan praised as most pragmatic.
